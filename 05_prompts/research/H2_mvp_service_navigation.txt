MVP Feasibility H2 — Feasible MVP for Resident Service Navigation

Define the feasible minimum viable prototype for the resident service navigation problem statement, given all known constraints.

CONSTRAINTS RECAP:
- No post-2018 RVA311 API — must build category taxonomy from manual research or historical Socrata data (2014–2015)
- rva.gov content must be crawled (no official data export)
- ~50+ RVA311 request types across 7 department groups
- Must not claim official City authority or simulate submissions
- Must include human fallback
- Must work on mobile

THREE APPROACHES TO EVALUATE:

A) Minimum Decision Tree (fastest to build):
- 5–8 questions covering the most common resident situations
- Covers 20–30 of the most common categories
- Hardcoded in JSON; no live data
- Time estimate: 8–12 hours for core logic and UI
- Demo: 5 common scenarios work reliably; edge cases acknowledged as "please call 311"

B) Search-Based Helper (more powerful):
- Crawl 50–100 rva.gov pages; build simple keyword/fuzzy search index
- Resident types description; tool returns top 3 category suggestions with explanations
- Time estimate: 12–16 hours
- Demo: Natural language query → ranked suggestions → official link

C) Hybrid (decision tree + search):
- Decision tree for known scenarios; search fallback for unknowns
- Time estimate: 16–20 hours (risky for 48-hour window)

For each approach: what is the minimum dataset required, what are the failure modes, and what does a "good enough for judges" demo look like?

Output
- Feasibility assessment for each approach
- Recommended approach with minimum viable feature set
- Go/no-go criteria (what data and logic must exist before the demo)
