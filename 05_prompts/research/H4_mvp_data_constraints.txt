MVP Feasibility H4 — Working Within the No-Post-2018-311-API Constraint

Investigate how a hackathon team can build a credible 311 service navigation tool given the single most important data constraint: the AvePoint Citizen Services platform (RVA311) has no public API and has not published data to Socrata since it launched in June 2018.

The constraint in detail:
- Current RVA311 data: completely inaccessible publicly
- Historical data: SeeClickFix era only (Jan 2014–Aug 2015, Socrata vgg4-hjn8)
- The category taxonomy in the current system may differ significantly from 2014–2015 categories
- Volume data (208,216 requests in 2025) is reported but not queryable

What substitutes can a team use to reconstruct the current 311 category taxonomy?
- Manual research on rva.gov (what service descriptions exist?)
- AvePoint Citizen Services product documentation (does it describe standard category structures?)
- Comparable city 311 category lists (NYC, Boston, SF) as proxies
- Open311 standard category definitions
- Historical Socrata data (how applicable is it to current categories?)

What can the historical data tell us that is still useful?
- Pattern of request types by season/geography
- Categories that were most common (likely still present)
- Geographic distribution of requests

How should a team communicate these constraints in the demo?
- What language correctly characterizes the tool as "based on public City content" rather than "using live 311 data"?
- How does this constraint affect the tool's accuracy claims?

Output
- Practical workarounds for building a category taxonomy without live API access
- Assessment of how much the 2014–2015 data helps vs. misleads
- Demo communication strategy for the data constraint
